## 🎯 项目愿景

**FormalMath** 致力于建立全球最大的开源数学教育知识库，推动数学知识的民主化、普及化和国际化。

### 核心使命

1. **教育普及**: 让高质量的数学教育资源对所有人免费开放
2. **知识整合**: 构建系统化、结构化的数学知识体系
3. **技术创新**: 探索数学教育与人工智能、形式化证明的融合
4. **社区共建**: 建立活跃、包容、可持续的开源社区

## 📜 治理原则

### 1. 开放性 (Openness)

- 所有决策过程对社区公开透明
- 欢迎任何人参与贡献和讨论
- 项目资源对所有人免费开放

### 2. 包容性 (Inclusivity)

- 尊重多元化背景和观点
- 为新手提供友好环境和支持
- 确保不同经验水平的人都能参与

### 3. 精英治理 (Meritocracy)

- 贡献质量和社区参与度决定影响力
- 基于能力和贡献授予责任和权限
- 建立公平的晋升和评价机制

### 4. 共识驱动 (Consensus-driven)

- 重要决策通过社区讨论达成共识
- 尊重少数意见，寻求最大公约数
- 在无法达成共识时，由核心团队做出最终决定

### 5. 责任与问责 (Accountability)

- 每个角色都承担相应责任
- 定期评估和反馈机制
- 对不当行为有明确的处理流程

## 👥 角色与职责

### 1. 项目创始人 (Project Founders)

**职责**:
- 制定项目愿景和核心方向
- 任命首届指导委员会成员
- 在重大分歧时做出最终决策

**产生方式**: 由项目创建者担任

**任期**: 终身（可主动退出）

---

### 2. 项目指导委员会 (Project Steering Committee, PSC)

#### 2.1 组成

- 3-7名成员
- 包括项目创始人、核心维护者代表、领域专家、社区代表
- 确保多样性和代表性

#### 2.2 职责

| 职责类别 | 具体内容 |
|:---------|:---------|
| 战略规划 | 制定长期发展规划、确定年度目标 |
| 政策制定 | 审批重要政策变更、治理规则修改 |
| 资源分配 | 预算分配、基础设施决策 |
| 冲突仲裁 | 处理重大社区冲突、决策僵局 |
| 品牌管理 | 项目品牌使用、对外合作授权 |
| 安全响应 | 重大安全事件的最终决策 |

#### 2.3 决策权限

- **全票通过**: 章程修改、项目方向重大变更
- **三分之二多数**: 新政策批准、预算审批
- **简单多数**: 日常事务决策

#### 2.4 产生方式

- 首届成员由创始人任命
- 后续通过委员会提名 + 社区投票
- 候选人需在项目贡献超过1年

#### 2.5 任期

- 任期: 2年
- 可连任一次，之后需间隔1年
- 每年改选约一半成员

---

### 3. 核心维护者 (Core Maintainers)

#### 3.1 数量

- 5-15名活跃维护者

#### 3.2 职责

| 职责类别 | 具体内容 |
|:---------|:---------|
| 代码/内容审核 | 审查 Pull Requests |
| 合并权限 | 将审核通过的 PR 合并到主分支 |
| 发布管理 | 版本发布、发布说明编写 |
| 基础设施 | CI/CD、服务器维护 |
| Issue 管理 | 分类、标记、关闭 Issues |
| 新成员指导 | 指导新贡献者、代码审查指导 |

#### 3.3 权限矩阵

| 权限 | 核心维护者 | 普通维护者 | 贡献者 |
|:-----|:----------:|:----------:|:------:|
| 代码合并 | ✅ | ⚠️ 需审核 | ❌ |
| 版本发布 | ✅ | ❌ | ❌ |
| 基础设施访问 | ✅ | ❌ | ❌ |
| 成员管理 | ✅ | ❌ | ❌ |
| Issue 标签 | ✅ | ✅ | ⚠️ 部分 |
| PR 审核 | ✅ | ✅ | ✅ |

#### 3.4 产生方式

- 由现有核心维护者提名
- 需在过去6个月内有显著贡献
- 需获得现有核心维护者三分之二同意

#### 3.5 任期

- 无固定任期
- 若6个月无活跃贡献，自动转为荣誉维护者
- 严重违反行为准则可被撤销

---

### 4. 领域专家 (Domain Experts)

#### 4.1 职责

- 审核特定数学分支的内容准确性
- 提供专业咨询和指导
- 参与相关领域的标准制定

#### 4.2 领域划分

| 领域 | MSC 范围 | 专家人数 |
|:-----|:---------|:--------:|
| 代数学 | 12-00, 13-00, 16-00, 18-00, 20-00 | 2-3 |
| 分析学 | 26-00, 30-00, 40-00, 46-00, 47-00 | 2-3 |
| 几何与拓扑 | 51-00, 53-00, 54-00, 55-00 | 2-3 |
| 数论 | 11-00 | 1-2 |
| 数理逻辑 | 03-00 | 1-2 |
| 应用数学 | 35-00, 65-00, 90-00 | 1-2 |
| 形式化证明 | Lean4, Coq 等 | 2-3 |

#### 4.3 产生方式

- 由项目指导委员会邀请
- 需提供专业背景证明
- 社区可推荐候选人

---

### 5. 社区代表 (Community Representatives)

#### 5.1 职责

- 收集和传达社区反馈
- 参与决策过程，代表普通贡献者声音
- 组织社区活动和讨论

#### 5.2 产生方式

- 每年通过社区投票选举
- 任何活跃贡献者都可参选或提名
- 任期1年，可连任

---

### 6. 活跃贡献者 (Active Contributors)

#### 6.1 定义

- 在过去3个月内至少有一次被合并的贡献
- 参与社区讨论和 Issue 回复
- 遵守社区行为准则

#### 6.2 权益

- 参与社区投票
- 申请成为维护者
- 参与政策讨论

---

### 7. 用户 (Users)

#### 7.1 定义

- 使用 FormalMath 资源的学习者
- 报告问题、提供反馈
- 帮助推广项目

#### 7.2 参与方式

- 通过 Issues 报告问题
- 参与 Discussions 讨论
- 在社交媒体上分享

## 📊 决策流程

### 1. 决策类型与流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        决策流程图                                │
└─────────────────────────────────────────────────────────────────┘

Level 1: 日常技术决策
├─ 范围: 代码审核、Bug修复、文档更新
├─ 决策者: 相关领域维护者
├─ 流程: PR 审核 → 合并
└─ 时效: 1-3天

Level 2: 功能/内容决策  
├─ 范围: 新功能、新文档分支、工具开发
├─ 决策者: 核心维护者共识
├─ 流程: Issue 讨论 → RFC → 投票 → 实施
└─ 时效: 1-2周

Level 3: 政策/流程决策
├─ 范围: 贡献指南变更、审核标准、晋升规则
├─ 决策者: 项目指导委员会
├─ 流程: 提案 → 社区讨论 → 委员会投票
└─ 时效: 2-4周

Level 4: 战略/治理决策
├─ 范围: 治理结构变更、重大方向调整、资金分配
├─ 决策者: 项目指导委员会(全票或三分之二)
├─ 流程: 提案 → 公开征求意见 → 委员会决策
└─ 时效: 1-2月
```

### 2. 提案流程 (RFC)

#### 2.1 RFC (Request for Comments) 流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   提交 RFC   │ -> │  社区讨论   │ -> │  修改完善   │
└─────────────┘    └─────────────┘    └──────┬──────┘
                                             │
              ┌──────────────────────────────┘
              │
              ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   投票决策   │ <- │  最终审查   │ <- │  修改完成   │
└─────────────┘    └─────────────┘    └─────────────┘
```

#### 2.2 RFC 模板

```markdown
# RFC: [标题]

**作者**: @username  
**日期**: YYYY-MM-DD  
**状态**: Draft / Discussion / Accepted / Rejected / Implemented  
**类型**: Feature / Policy / Process  
**影响范围**: Code / Documentation / Community

## 摘要
一句话概括提案内容。

## 动机
为什么需要这个变更？解决什么问题？

## 详细设计
具体实施方案。

## 替代方案
考虑过哪些其他方案？

## 影响分析
- 对用户的影响
- 对贡献者的影响
- 对维护者的影响

## 时间线
建议的实施时间表。
```

### 3. 投票机制

#### 3.1 投票类型

| 类型 | 适用场景 | 通过门槛 | 参与者 |
|:-----|:---------|:---------|:-------|
| 简单多数 | 日常决策 | >50% | 相关人员 |
| 三分之二 | 重要变更 | ≥66.7% | 核心维护者 |
| 全票通过 | 章程修改 | 100% | 指导委员会 |
| 共识决策 | 社区事务 | 无反对票 | 活跃贡献者 |

#### 3.2 投票方式

- **GitHub Reaction**: 快速意见收集
- **评论投票**: +1 / -1 及理由
- **正式投票**: 使用投票工具，匿名/实名

#### 3.3 投票期限

- 一般决策: 7天
- 重要决策: 14天
- 紧急决策: 缩短至3天，需说明原因

## ⚖️ 冲突解决机制

### 1. 冲突类型

#### 1.1 技术冲突

- 实现方案分歧
- 架构设计争议
- 工具选择争议

#### 1.2 内容冲突

- 数学内容准确性争议
- 教学顺序和方法分歧
- 术语使用争议

#### 1.3 人际冲突

- 沟通方式不当
- 行为准则违反
- 贡献归属争议

### 2. 冲突解决流程

```
Level 1: 直接沟通 (当事人)
├─ 私下友好讨论
├─ 寻求共识
└─ 期望解决: 80%的冲突

Level 2: 调解 (中立第三方)
├─ 请求社区代表或维护者介入
├─ 了解双方观点
├─ 提出调解方案
└─ 期望解决: 15%的冲突

Level 3: 正式仲裁 (核心团队)
├─ 提交至指导委员会
├─ 正式调查
├─ 做出约束性决定
└─ 期望解决: 4%的冲突

Level 4: 项目指导委员会终审
├─ 极少数重大冲突
├─ 全委员会审议
├─ 最终决定
└─ 期望解决: 1%的冲突
```

### 3. 调解指南

#### 3.1 调解原则

- **中立性**: 调解人不偏袒任何一方
- **保密性**: 尊重当事人隐私
- **及时性**: 尽快介入，防止升级
- **建设性**: 寻求双赢解决方案

#### 3.2 调解步骤

1. **了解情况**: 分别听取双方陈述
2. **明确问题**: 确定冲突核心
3. **探索方案**: 寻找可能的解决方案
4. **达成共识**: 促使双方接受方案
5. **跟进落实**: 确保方案执行

### 4. 行为准则执行

#### 4.1 违规处理流程

| 严重程度 | 处理方式 | 决策者 |
|:---------|:---------|:-------|
| 轻微违规 | 私下警告、教育 | 维护者 |
| 一般违规 | 公开警告、临时禁言 | 核心维护者 |
| 严重违规 | 长期禁言、封禁 | 指导委员会 |
| 极端违规 | 永久封禁、法律行动 | 指导委员会 |

#### 4.2 申诉机制

- 当事人可在7天内提出申诉
- 指导委员会进行复核
- 复核决定为最终决定

## 📋 政策制定与修改

### 1. 可修改的政策

| 政策文件 | 修改权限 | 通过门槛 |
|:---------|:---------|:---------|
| 社区治理模式 | 指导委员会 | 全票 |
| 行为准则 | 指导委员会 | 三分之二 |
| 贡献指南 | 核心维护者 | 三分之二 |
| 技术规范 | 核心维护者 | 简单多数 |
| 内容标准 | 领域专家 | 简单多数 |

### 2. 修改流程

1. **提案**: 提交 RFC
2. **讨论**: 社区公开讨论至少14天
3. **修订**: 根据反馈修改
4. **投票**: 相关决策机构投票
5. **实施**: 通过后更新文档并公告
6. **审查**: 6个月后评估效果

### 3. 政策版本管理

- 所有政策文件版本化
- 保留修改历史
- 重大变更需发布迁移指南

## 🔍 透明度与问责制

### 1. 信息公开

#### 1.1 必须公开的信息

| 信息类型 | 公开方式 | 更新频率 |
|:---------|:---------|:---------|
| 治理结构 | 文档 | 实时 |
| 决策记录 | GitHub Issues | 实时 |
| 会议纪要 | 文档/讨论区 | 每月 |
| 财务报告 | 文档 | 每季度 |
| 活动报告 | 博客 | 每月 |

#### 1.2 会议透明度

- 指导委员会会议记录公开
- 核心维护者会议摘要公开
- 敏感信息安全处理

### 2. 问责机制

#### 2.1 定期评估

| 评估对象 | 评估方式 | 频率 |
|:---------|:---------|:-----|
| 指导委员会 | 社区满意度调查 | 每年 |
| 核心维护者 | 贡献统计 + 同行评价 | 每半年 |
| 治理效果 | 社区健康度指标 | 每季度 |

#### 2.2 问责措施

- 表现不佳的角色成员可被提醒
- 持续不履职可被撤销资格
- 重大过失需公开说明

### 3. 社区反馈渠道

| 渠道 | 用途 | 响应时间 |
|:-----|:-----|:---------|
| GitHub Discussions | 一般讨论 | 3天 |
| GitHub Issues | 政策建议 | 7天 |
| 邮件列表 | 正式投诉 | 7天 |
| 匿名表单 | 敏感反馈 | 14天 |

### 3. 经验总结

- 定期总结治理经验教训
- 学习其他成功开源项目
- 持续改进治理模式
