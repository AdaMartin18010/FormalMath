## 详细流程说明

### 1. 反馈提交阶段

#### 1.1 提交方式
- **API提交**：通过 `/api/v1/feedback/feedbacks` 接口
- **匿名提交**：`user_id` 可为空
- **关联信息**：可关联概念ID、页面URL、会话ID

#### 1.2 收集信息
```json
{
  "title": "反馈标题",
  "content": "详细描述",
  "browser_info": {
    "browser": "Chrome",
    "version": "120.0",
    "os": "Windows 10",
    "user_agent": "..."
  },
  "device_info": {
    "type": "desktop",
    "screen": "1920x1080"
  },
  "session_id": "sess_xxx",
  "satisfaction_rating": 4
}
```

#### 1.3 异步处理
提交后立即返回，后台异步处理分类任务

---

### 2. 自动分类阶段

#### 2.1 分类算法
使用基于规则的分类器（`FeedbackClassifier`）

**分类维度：**
- **反馈类型**：8种类型（bug_report, feature_request等）
- **优先级**：critical, high, medium, low
- **情感倾向**：positive, negative, neutral

#### 2.2 分类规则
```python
# 关键词匹配规则
KEYWORD_RULES = {
    FeedbackType.BUG_REPORT: [
        'bug', '错误', '崩溃', '闪退', '异常', '报错',
        'crash', 'freeze', 'broken', 'not working'
    ],
    FeedbackType.FEATURE_REQUEST: [
        '功能', '建议', '增加', '希望', '想要',
        'feature', 'request', 'suggest', 'improvement'
    ],
    # ... 其他类型
}
```

#### 2.3 置信度计算
```
confidence = 0.5 + (最佳类型得分 / 总得分) × 0.5
```
- 置信度 > 0.8：高可信度
- 置信度 0.5-0.8：中等可信度
- 置信度 < 0.5：低可信度，需要人工确认

```
    │  closed │◄─────────────────────│ resolved │
    │(自动关闭)│      7天后            │          │
    └─────────┘                      └──────────┘
```

#### 状态说明

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| `pending` | 待处理 | 用户提交后初始状态 |
| `classified` | 已分类 | 自动分类完成后 |
| `under_review` | 审核中 | 管理员开始审核或添加回复 |
| `in_progress` | 处理中 | 管理员确认接受并开始处理 |
| `resolved` | 已解决 | 问题解决 |
| `closed` | 已关闭 | 解决7天后自动关闭或被拒绝 |
| `rejected` | 已拒绝 | 管理员拒绝处理 |

---

### 4. 通知机制

#### 4.1 通知渠道
根据优先级选择通知渠道：

| 优先级 | 通知渠道 |
|--------|----------|
| `critical` | 邮件 + Slack + 短信 |
| `high` | 邮件 + Slack |
| `medium/low` | 邮件 |
| 其他 | 日志记录 |

#### 4.2 通知内容模板
```
【新反馈通知】

类型: {feedback_type}
优先级: {priority}
标题: {title}
提交时间: {created_at}

查看详情: https://admin.example.com/feedback/{id}[需更新]
```

---

### 5. 管理员处理流程

#### 5.1 仪表板工作流
1. **查看仪表板**：获取待处理反馈概览
2. **筛选优先级**：优先处理 critical/high
3. **查看详情**：了解反馈完整内容
4. **确认分类**：必要时调整类型和优先级
5. **分配给处理人**：指定负责人

#### 5.2 处理操作
- **接受**：状态更新为 `in_progress`
- **需要补充**：添加回复要求用户提供更多信息
- **转交**：更改 `assigned_to` 转给其他团队
- **拒绝**：状态更新为 `rejected`，说明原因

---

### 6. 回复管理

#### 6.1 回复类型
- **公开回复**：用户可见，发送通知给用户
- **内部备注**：仅管理员可见，用于内部沟通

#### 6.2 回复流程
```
添加回复
    │
    ├─► 公开回复 ──► 发送用户通知 ──► 更新状态为 under_review
    │
    └─► 内部备注 ──► 仅记录日志
```

---

### 7. 解决与关闭

#### 7.1 标记解决
```python
# 更新反馈状态
service.update_feedback(
    feedback_id=123,
    status=FeedbackStatus.RESOLVED,
    resolution_notes="问题已修复，原因是..."
)
```

#### 7.2 自动关闭
Celery定时任务每天检查：
- 已解决超过7天的反馈
- 自动更新状态为 `closed`
- 记录系统自动关闭日志

---

### 8. 统计分析

#### 8.1 统计维度
- **时间维度**：日/周/月报
- **类型维度**：各类型反馈数量
- **状态维度**：处理效率分析
- **满意度维度**：用户评价分布

#### 8.2 定时任务
| 任务 | 频率 | 说明 |
|------|------|------|
| `process_new_feedback` | 每小时 | 处理新反馈 |
| `auto_close_resolved` | 每天 | 自动关闭已解决反馈 |
| `generate_daily_analytics` | 每天 | 生成日报 |
| `generate_weekly_analytics` | 每周 | 生成周报 |
| `generate_monthly_analytics` | 每月 | 生成月报 |

## API 使用场景

### 场景1：用户提交反馈

```python
import requests

# 用户发现搜索功能有问题
response = requests.post(
    "https://api.example.com/api/v1/feedback/feedbacks[需更新]",
    json={
        "title": "搜索功能无法使用",
        "content": "在搜索框输入关键词后，页面一直显示加载中",
        "related_page": "/search",
        "browser_info": {
            "browser": "Chrome",
            "version": "120.0"
        }
    }
)

# 系统自动分类为 bug_report，优先级 high
result = response.json()
# {
#   "success": True,
#   "data": {
#     "id": 123,
#     "feedback_type": "bug_report",
#     "status": "pending",
#     "priority": "high"
#   }
# }
```

### 场景2：管理员处理反馈

```python

    "https://api.example.com/api/v1/feedback/feedbacks[需更新]",
    params={
        "status": "pending",
        "priority": "high",

# 2. 查看反馈详情

    "https://api.example.com/api/v1/feedback/feedbacks/123[需更新]"

    "https://api.example.com/api/v1/feedback/feedbacks/123[需更新]",

        "status": "in_progress",
        "assigned_to": 5,  # 开发人员ID
        "resolution_notes": "已定位问题，正在修复"
    }
)

# 4. 添加回复通知用户
requests.post(
    "https://api.example.com/api/v1/feedback/feedbacks/123/responses[需更新]",
    json={
        "content": "感谢您的反馈，问题已修复，请刷新页面重试",
        "responder_id": 5,
        "is_internal": False
    }
)

# 5. 标记已解决
requests.put(
    "https://api.example.com/api/v1/feedback/feedbacks/123[需更新]",
    json={
        "status": "resolved",
        "resolution_notes": "修复了搜索API的超时问题"
    }
)
```

## 最佳实践

### 1. 反馈提交
- 提供清晰的分类选项，但允许用户不选择（自动分类）
- 收集必要的环境信息帮助复现问题
- 设置合理的字符限制（标题255，内容5000）

### 2. 自动分类优化
- 定期分析分类准确率，更新关键词规则
- 对低置信度（<0.7）的反馈优先人工审核
- 收集管理员修正数据用于改进算法

### 3. 处理效率
- 设置优先级规则，确保紧急问题及时处理
- 建立反馈升级机制，超时未处理的自动升级
- 定期分析处理时间，识别瓶颈

### 4. 用户沟通
- 对每个反馈都给予响应，即使是拒绝也要说明原因
- 公开回复保持专业友好的语气
- 问题解决后主动通知用户

### 5. 数据分析
- 定期分析反馈类型分布，发现产品改进点
- 关注满意度趋势，持续优化用户体验
- 监控处理效率指标，优化工作流程

## 扩展建议

### 1. 集成AI模型
- 使用BERT等预训练模型进行更准确的分类
- 实现智能回复建议
- 情感分析精细化

### 2. 多语言支持
- 支持多语言反馈的自动检测和分类
- 机器翻译辅助理解

### 3. 用户反馈闭环
- 问题解决后邀请用户确认
- 收集用户对处理结果的满意度
- 建立积分奖励机制鼓励反馈

### 4. 高级分析
- 反馈热力图（按功能模块）
- 用户反馈行为画像
- 预测性分析（提前发现潜在问题）
