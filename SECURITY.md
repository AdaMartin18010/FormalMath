## 报告安全漏洞

### 负责任的披露

我们非常重视安全问题。如果您发现了安全漏洞，请按照以下流程负责任地披露：

1. **不要** 在公开的 Issue 或 Discussion 中披露漏洞
2. **通过私密渠道** 向维护者报告
3. **给予我们合理的时间** 来修复问题后再公开

### 报告方式

请通过以下方式之一报告安全问题：

| 方式 | 联系方式 | 响应时间 |
|------|----------|----------|
| GitHub 安全建议 | [Security Advisories](../../security/advisories) | 48小时内 |
| 私密邮件 | security@formalmath.example.com | 48小时内 |

### 报告内容

请尽可能包含以下信息：

- **漏洞描述**：清晰描述漏洞的性质
- **影响范围**：哪些版本/组件受到影响
- **复现步骤**：详细的复现步骤
- **可能的修复建议**（可选）
- **您的联系方式**：以便我们与您沟通

## 安全更新流程

### 处理流程

```
收到报告 → 确认收到 → 评估影响 → 开发修复 → 测试验证 → 发布更新 → 公开披露
```

### 时间线

| 阶段 | 目标时间 |
|------|----------|
| 确认收到 | 48小时内 |
| 初步评估 | 7天内 |
| 修复开发 | 根据严重程度 |
| 发布更新 | 修复完成后尽快 |

### 严重程度分级

| 级别 | 描述 | 响应时间 |
|------|------|----------|
| 🔴 严重 | 数据泄露、系统接管 | 24小时内 |
| 🟠 高 | 权限提升、敏感信息泄露 | 72小时内 |
| 🟡 中 | 功能异常、性能影响 | 14天内 |
| 🟢 低 | 轻微问题 | 下次更新 |

## 安全最佳实践

### 对于贡献者

- **保护凭据**：不要将密码、API密钥等敏感信息提交到仓库
- **依赖审查**：更新依赖时审查安全性
- **代码审查**：所有代码变更应经过审查
- **最小权限**：仅访问必要的资源

### 对于用户

- **保持更新**：及时更新到最新版本
- **审查依赖**：了解项目依赖的其他组件
- **报告问题**：发现可疑行为请及时报告

## 安全相关配置

### 自动化安全检查

项目使用 GitHub Actions 进行自动化安全检查：

- **依赖漏洞扫描**：定期扫描依赖项中的已知漏洞
- **密钥检测**：防止意外提交敏感信息
- **代码分析**：静态代码安全分析

工作流程文件：
- `.github/workflows/security.yml`
