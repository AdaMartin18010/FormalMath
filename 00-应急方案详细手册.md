## 应急响应流程

### 整体流程图

```
发现故障
    ↓
初步评估（5分钟内）
    ↓
┌─────────────────┬─────────────────┐
↓                 ↓                 ↓
P0-紧急          P1-严重           P2/P3-一般
立即回滚         评估后回滚         热修复
    ↓                 ↓                 ↓
恢复服务          恢复服务          恢复服务
    ↓                 ↓                 ↓
验证修复          验证修复          验证修复
    ↓                 ↓                 ↓
通知用户          通知用户          记录问题
    ↓                 ↓                 ↓
事后复盘          事后复盘          排期修复
```

### 时间线要求

| 时间 | 动作 | 负责人 |
|------|------|--------|
| T+0 | 故障发现（监控/用户反馈）| 监控/客服 |
| T+5min | 初步评估，确定级别 | 运维 |
| T+10min | P0/P1故障：开始回滚 | 技术负责人 |

| T+24h | 完整事故报告 | 技术负责人 |

## 故障诊断手册

### 2.1 网站无法访问

**检查步骤**:

```bash
# 1. 检查域名解析
nslookup formalmath.org
dig formalmath.org

# 2. 检查网络连通性
ping formalmath.org
traceroute formalmath.org

# 3. 检查HTTP响应
curl -I https://formalmath.org
curl -s -o /dev/null -w "%{http_code}" https://formalmath.org

# 4. 检查CDN状态
# CloudFlare状态页: https://www.cloudflarestatus.com/

# 5. 检查服务器状态
kubectl get pods -n formalmath
kubectl describe pod <pod-name> -n formalmath

# 6. 检查负载均衡器
kubectl get svc -n formalmath
kubectl describe svc formalmath-frontend -n formalmath
```

**常见问题**:

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| DNS解析失败 | DNS配置错误/过期 | 检查DNS记录，更换DNS服务器 |
| 连接超时 | 服务器宕机/网络中断 | 检查服务器状态，联系云服务商 |
| 502 Bad Gateway | 后端服务崩溃 | 重启后端服务 |
| 503 Service Unavailable | 服务过载 | 扩容/限流/降级 |
| SSL证书错误 | 证书过期/配置错误 | 更新证书 |
| 404大量出现 | 路由配置错误/文件丢失 | 检查部署完整性 |

### 2.2 页面加载缓慢

**诊断命令**:

```bash
# 1. 检查前端资源加载
curl -s -w "\nDNS: %{time_namelookup}s\nConnect: %{time_connect}s\nTotal: %{time_total}s\n" \
  https://formalmath.org

# 2. 检查服务器资源
top -p $(pgrep -d',' node)
df -h
free -m

# 3. 检查数据库性能
# PostgreSQL慢查询
SELECT query, mean_time, calls 
FROM pg_stat_statements 
ORDER BY mean_time DESC 
LIMIT 10;

# 4. 检查API响应时间
# 使用apm工具或日志分析

# 5. 检查CDN命中率
curl -I https://formalmath.org/static/main.js | grep -i cache
```

**优化措施**:

1. **启用压缩**: 确保gzip/brotli已启用
2. **优化图片**: 转换为WebP，使用响应式图片
3. **代码分割**: 懒加载非关键JS/CSS
4. **缓存优化**: 调整Cache-Control策略
5. **数据库优化**: 添加索引，优化慢查询

### 2.3 API错误率高

**诊断步骤**:

```bash
# 1. 查看错误日志
kubectl logs -f deployment/api -n formalmath | grep ERROR

# 2. 检查错误类型分布
grep "ERROR" /var/log/formalmath/api.log | awk '{print $5}' | sort | uniq -c | sort -rn

# 3. 检查依赖服务状态
# 数据库
curl -s http://localhost:5432/health || echo "DB unreachable"

# Redis
redis-cli ping

# 4. 检查资源限制
kubectl describe pod <api-pod> -n formalmath | grep -A5 "Limits"

# 5. 检查并发连接数
ss -s
netstat -an | grep :8080 | wc -l
```

**常见API错误**:

| 错误码 | 原因 | 处理 |
|--------|------|------|
| 429 Too Many Requests | 限流触发 | 检查限流配置，考虑扩容 |
| 500 Internal Error | 代码异常 | 查看堆栈跟踪，热修复或回滚 |
| 502/503 | 服务不可用 | 检查后端健康状态 |
| 504 Gateway Timeout | 上游超时 | 检查数据库/外部API |

### 2.4 数据库问题

#### 连接池耗尽

```sql
-- 检查当前连接数
SELECT count(*) FROM pg_stat_activity;

-- 检查连接状态
SELECT state, count(*) FROM pg_stat_activity GROUP BY state;

-- 检查空闲连接
SELECT pid, usename, application_name, state, state_change 
FROM pg_stat_activity 
WHERE state = 'idle' AND state_change < NOW() - INTERVAL '1 hour';

-- 终止空闲连接
SELECT pg_terminate_backend(pid) 
FROM pg_stat_activity 
WHERE state = 'idle' AND state_change < NOW() - INTERVAL '1 hour';
```

#### 磁盘空间不足

```bash
# 检查磁盘使用
df -h

# 检查PostgreSQL数据目录大小
du -sh /var/lib/postgresql/data

# 检查WAL文件
ls -la /var/lib/postgresql/data/pg_wal/

# 清理策略
# 1. 清理日志
find /var/log/postgresql -name "*.log" -mtime +7 -delete

# 2. 清理临时文件
VACUUM FULL;

# 3. 如仍不足，扩容磁盘
```

#### 主从同步延迟

```sql
-- 在主库上
SELECT pg_current_wal_lsn();

-- 在从库上  
SELECT pg_last_wal_receive_lsn(), pg_last_wal_replay_lsn();

-- 计算延迟（字节）
SELECT pg_current_wal_lsn() - pg_last_wal_replay_lsn() AS lag_bytes;
```

### 2.5 Lean4编译失败

```bash
# 1. 检查Lean版本
lean --version

# 2. 检查mathlib4版本
cat lake-manifest.json | grep -A2 mathlib

# 3. 清理并重新构建
lake clean
rm -rf .lake
lake update
lake build

# 4. 检查错误日志
lake build 2>&1 | tee build.log
grep "error:" build.log

# 5. 如mathlib更新导致不兼容，回滚到稳定版本
git checkout v4.29.0 -- lake-manifest.json
lake update
lake build
```

### 3.3 数据库服务

**部署架构**:
- PostgreSQL 15
- 主从复制（1主2从）
- 自动故障转移（Patroni）

**故障场景**:

1. **主库故障**
   ```bash
   # Patroni会自动提升从库
   # 检查集群状态
   patronictl -c /etc/patroni.yml list
   
   # 手动切换（如自动切换失败）
   patronictl -c /etc/patroni.yml switchover
   
   # 修复原主库
   # 作为从库重新加入集群
   patronictl -c /etc/patroni.yml reinit formalmath <node_name>
   ```

2. **从库延迟过大**
   ```sql
   -- 在从库上取消长时间运行的查询
   SELECT pg_terminate_backend(pid) 
   FROM pg_stat_activity 
   WHERE state = 'active' AND now() - query_start > interval '1 hour';
   ```

3. **表损坏**
   ```bash
   # 停止写入
   # 使用pg_dump备份
   pg_dump -h localhost -U formalmath formalmath_db > backup.sql
   
   # 使用pg_resetwal（极端情况）
   pg_resetwal -D /var/lib/postgresql/data
   
   # 从备份恢复
   pg_restore -h localhost -U formalmath -d formalmath_db backup.sql
   ```

## 数据恢复手册

### 4.1 备份策略

| 数据类型 | 备份方式 | 频率 | 保留期 | 存储位置 |
|----------|----------|------|--------|----------|
| PostgreSQL | pg_dump + WAL归档 | 每日全量 + 持续增量 | 30天 | 云存储 |
| Redis | RDB + AOF | 每小时 | 7天 | 本地+云 |
| 文档文件 | rclone同步 | 实时 | 90天 | 多区域云存储 |
| 配置文件 | Git + 定期导出 | 每次变更 | 无限 | Git仓库 |

### 4.2 恢复步骤

#### 场景：数据库误删除

```bash
# 1. 立即停止写入
kubectl scale deployment api --replicas=0 -n formalmath

# 2. 确定恢复时间点
# 查看备份列表
aws s3 ls s3://formalmath-backups/postgresql/ --recursive | grep "$(date +%Y-%m-%d)"

# 3. 下载备份
aws s3 cp s3://formalmath-backups/postgresql/backup_$(date +%Y%m%d).sql.gz ./
gunzip backup_*.sql.gz

# 4. 创建新数据库
createdb -h localhost -U formalmath formalmath_db_new

# 5. 恢复数据
pg_restore -h localhost -U formalmath -d formalmath_db_new backup_*.sql

# 6. 验证数据完整性
psql -h localhost -U formalmath -d formalmath_db_new -c "SELECT COUNT(*) FROM documents;"

# 7. 切换数据库连接
# 更新环境变量，重启服务
kubectl set env deployment/api DATABASE_URL=postgresql://.../formalmath_db_new -n formalmath

# 8. 恢复服务
kubectl scale deployment api --replicas=3 -n formalmath

# 9. 验证
kubectl logs -f deployment/api -n formalmath | grep -i error
```

#### 场景：点-in-time恢复

```bash
# 使用WAL归档进行PITR
# 1. 准备基础备份
# 2. 提取需要的WAL文件
# 3. 恢复到指定时间点

# pgBackRest示例
pgbackrest restore --stanza=formalmath --type=time \
  --target="2026-04-04 14:30:00" \
  --target-action=promote
```

### 4.3 数据一致性校验

```sql
-- 文档数量校验
SELECT COUNT(*) as total_docs FROM documents;
-- 预期值：6573

-- 关键表记录数
SELECT 
  (SELECT COUNT(*) FROM mathematicians) as mathematicians,
  (SELECT COUNT(*) FROM concepts) as concepts,
  (SELECT COUNT(*) FROM courses) as courses;

-- 索引完整性
SELECT schemaname, tablename, indexname 
FROM pg_indexes 
WHERE schemaname = 'public';

-- 外键约束检查
SELECT 
  tc.constraint_name, 
  tc.table_name, 
  kcu.column_name,
  ccu.table_name AS foreign_table_name,
  ccu.column_name AS foreign_column_name 
FROM information_schema.table_constraints AS tc 
JOIN information_schema.key_column_usage AS kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage AS ccu
  ON ccu.constraint_name = tc.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY';
```
